## 二、根因分析

### 2.1 直接原因

**SQLite Python 的默认 `isolation_level=''` 开启了隐式事务机制。**

当 `isolation_level=''`（Python sqlite3 模块的默认值）时，连接对象在执行第一个写操作（DML）时会自动执行 `BEGIN` 语句获取 RESERVED 锁。只要不显式调用 `commit()` 或 `rollback()`，事务就保持开放状态。

在 DELETE 和 WAL 两种 journal 模式下，这个行为的不同表现形式：

**DELETE 模式下：**
- 事务一旦 `BEGIN` 获取 RESERVED 锁，创建 `.journal` 文件
- 只要事务未提交，journal 文件持续存在
- 后续所有写操作尝试获取写锁时，检测到 journal 文件存在 → 等待 `busy_timeout`（60秒）→ 超时抛出 `OperationalError: database is locked`

**WAL 模式下：**
- 事务 `BEGIN` 后创建 WAL 文件（有时是 SHM 文件）
- 同样存在隐式事务持有锁的问题
- 此外 WAL 的多线程读特性会与 APScheduler 的 ThreadPoolExecutor 产生写锁竞争

### 2.2 为什么主线程的连接持锁不释放？

Flask（threaded=True）模式下，每个 HTTP 请求由独立线程处理。`get_db()` 使用 `threading.local()` 为每个线程缓存独立连接。但关键问题在于：

1. 某个 HTTP 请求线程（如登录请求）执行了写操作（更新 session 状态）
2. 由于 `isolation_level=''`，连接自动开启了隐式事务
3. 该事务在函数返回时**没有显式 `commit()`**（Flask sqlite3 默认行为）
4. 连接被放回 thread-local 缓存（`_local.conn`），连接对象没有被 close，**事务状态保持**
5. journal 文件永久存在于磁盘
6. APScheduler 的 ThreadPoolExecutor-0_0 线程尝试写操作 → 检测 journal 文件 → 等待 60 秒 → 超时

### 2.3 为什么之前正常运行？

根据 app.log，问题是 2026-05-29 16:09 之后才开始出现的。这很可能是因为在那之前：
- 没有触发写操作密集的场景
- 或者某次 HTTP 请求的写操作路径改变了事务提交时机
- 与 `isolation_level=''` 相关的行为在不同代码路径下表现不一致

### 2.4 为什么 journal 文件在进程 kill 后仍然存在？

这是最关键的诊断线索。journal 文件在进程异常终止后仍然存在，证明 SQLite 层有一个持久的未提交事务正在活跃。这不是进程崩溃导致的残留，而是活跃进程持续持有事务的状态。

## 三、修复方案

### 3.1 核心修复：autocommit 模式

```python
# 修复前
_local.conn = sqlite3.connect(DB_PATH, check_same_thread=False)
_local.conn.execute("PRAGMA journal_mode=WAL")

# 修复后
_local.conn = sqlite3.connect(DB_PATH, check_same_thread=False, isolation_level=None)
_local.conn.execute("PRAGMA journal_mode=DELETE")
_local.conn.execute("PRAGMA busy_timeout=60000")
```

`isolation_level=None` 启用 SQLite 的 autocommit 模式：**每个 SQL 语句执行后立即自动提交，不存在事务残留**。任何写操作都是独立的最小事务，写完即释放锁。

### 3.2 journal mode 改为 DELETE

WAL 模式在多线程写场景下有固有的锁竞争问题（WAL-1 写锁与多读锁的冲突）。DELETE 模式在单进程写场景下更简单可靠，无需维护 WAL checkpoint daemon。

### 3.3 移除 WAL checkpoint daemon

```python
# 已移除 _wal_checkpoint_loop daemon 线程
# DELETE 模式不需要定期 PRAGMA wal_checkpoint(PASSIVE)
```

## 五、教训与防范建议

### 5.1 教训

1. **SQLite Python 默认的隐式事务是陷阱**：任何使用 Python sqlite3 模块的服务，只要开启了多线程写（包括 HTTP 请求线程 + 后台调度线程），就必须显式设置 `isolation_level=None`（autocommit）或始终在写操作后显式 `commit()`。默认的隐式事务在某些边缘场景下会导致事务未提交。

2. **journal 文件不消失是最强的诊断信号**：进程存活期间，任何时候 `.journal`（或 `.wal`/`.shm`）文件存在，都说明有未提交的事务。这是 SQLite 层面的告警，不能忽略。

3. **60 秒 busy_timeout 是双刃剑**：60 秒的超时保护防止了立即失败，但也导致每次失败需要等待 60 秒才暴露问题。如果超时设为 5 秒，问题会更早被发现。

### 5.2 防范建议

1. **代码规范**：在所有 `sqlite3.connect()` 调用处强制使用 `isolation_level=None`，在代码审查中明确检查 journal_mode 配置。

2. **监控指标**：增加 journal 文件存在性检测（crontab 或后台线程定期检查），发现 `.journal`/`.wal`/`.shm` 文件立即告警。

3. **连接管理**：对 `get_db()` 返回的连接增加"检查点"——每次获取连接后立即检查是否有未关闭的事务（通过 `in_transaction` 属性），如有则强制 `rollback()`。
