# 订阅条件细化设计方案

**状态**: 已讨论，待实施
**日期**: 2026-05-23

---

## 一、现状

当前 `filter_conditions` 结构：

```json
{
  "all_products": false,
  "chains": [
    ["WAF", "V5.5", "规则库升级包"],
    ["WAF", "V5.5", "规则库升级包", "http://..."]
  ]
}
```

**问题**：
- `chains` 存全路径（产品→版本→类型→URL），全路径匹配太细
- 想订阅"WAF V5.5 所有类型"必须列出每个类型，无法泛化

---

## 二、目标

支持任意层级选择：产品 / 产品+版本 / 产品+版本+类型

---

## 三、新结构设计

```json
{
  "all_products": false,
  "filters": [
    { "p": "WAF", "v": "V5.5", "t": ["规则库升级包", "产品完整包"] },
    { "p": "WAF", "v": "V5.6", "t": ["*"] },
    { "p": "IPS", "v": "*", "t": ["*"] }
  ]
}
```

**语义**：
- `p="*"` = 所有产品
- `v="*"` = 该产品下所有版本
- `t=["*"]` = 该版本下所有类型
- `t=["规则库升级包"]` = 精确匹配该类型

---

## 四、匹配逻辑

当新包到达时：

```
1. all_products=true → 匹配所有
2. 遍历 filters，找 p==product_name
   3. v=="*" 或 v==version_branch → 匹配该版本
      4. t 包含 "*" → 匹配该版本所有类型
      5. t 为具体类型列表 → package_type 必须在列表中
5. 无匹配 → 不推送
```

---

## 五、兼容性

| 场景 | 处理 |
|------|------|
| 读取旧 `chains` 格式 | 自动转换为新 `filters` 格式 |
| 写入新规则 | 统一存为新格式 |
| schema | 不变（TEXT(JSON)），无需迁移 |

---

## 六、后端改动点

| 文件 | 改动 |
|------|------|
| `subscription.py` | 读写增加版本兼容层（chains ↔ filters） |
| `router.py` `get_new_for_subscription()` | chain 全路径比较 → 前缀树匹配 |
| `change.py` | 同上匹配逻辑 |

---

## 七、UI 改动

复用现有 `openRModal` 四区弹窗：

- **左侧树**：按产品→版本→类型展开，支持多选 checkbox
- **右侧已选**：显示 `{产品} > {版本} > {类型}` pills
- **中间层选择**：
  - 选产品节点 → 包含所有版本和类型
  - 选版本节点 → 包含所有类型
  - 选类型节点 → 精确匹配
