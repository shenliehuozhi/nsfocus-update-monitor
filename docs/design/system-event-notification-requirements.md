# 系统事件通知需求文档

> 版本：v1.0
> 日期：2026-05-24
> 状态：草稿

---

## 1. 背景与目标

### 1.1 背景

当前订阅规则系统面向"绿盟产品版本更新"场景，触发机制为拉（定时扫描）。缺乏系统自身运行状态的监控告警能力，管理员无法及时发现采集器异常、推送失败、日志报错等问题。

### 1.2 目标

为系统提供全方位的可观测性，覆盖采集健康、通知健康、规则健康、系统健康四大维度。系统事件统一通过专用通知渠道推送给运维/SRE人员，实现故障早发现、早处理。

---

## 2. 核心设计决策

### 2.1 独立于订阅规则

系统事件通知不依附于订阅规则，而是作为独立的全局配置存在于「系统设置」中。

| 对比维度 | 订阅规则（产品更新） | 系统设置（系统事件） |
|---------|-------------------|-------------------|
| 作用域 | 每个客户/规则独立 | 全局统一 |
| 条件复杂度 | 路径树（多层级） | 事件类型多选（简单） |
| 触发方式 | 拉（定时扫描） | 推（事件驱动） |
| 维护者 | 客户管理员 | 系统管理员 |
| 通知受众 | 客户/业务负责人 | 运维/SRE |
| SLA要求 | 低（仅供参考） | 高（影响业务） |
| 推送模式 | 即时/汇总均可 | 实时优先，周期可配 |

### 2.2 独立通知渠道

系统事件与产品更新使用**独立通知渠道**，避免重要告警淹没在日常更新推送中。

## 3. 事件分类与定义

### 3.1 P0 - 核心可观测性（必须实现）

| 事件类型 | 触发条件 | 推送模式 | 说明 |
|---------|---------|---------|------|
| 采集失败 | 任意单个URL采集失败 | 实时 | 不要求全部失败才算 |
| 推送失败 | 发送通知失败（含包撤回） | 实时 | 包被撤回视为推送失败 |
| 规则无匹配 | 连续3个周期无匹配 | 汇总 | 可能路径配置错误，需关注 |
| 规则过期 | valid_until到达 | 汇总 | 推送已停止，需立即处理 |
| 新包类型发现 | 发现新的product/version/type组合 | 汇总 | 路径树扩展了，管理员需知 |
| 服务重启 | 进程异常退出后恢复 | 实时 | 需区分手动重启与异常崩溃 |

**推送模式说明**：
- **实时**：事件发生立即推送（如采集失败、服务重启）
- **汇总**：按天/周期聚合后推送（如规则无匹配、规则过期）

### 3.2 P1 - 系统健康（第二期）

| 事件类型 | 触发条件 | 严重程度 | 说明 |
|---------|---------|---------|------|
| 磁盘空间低 | 使用率超过85% | 高 | 影响数据写入 |
| 数据库异常 | 连接失败/SQL错误 | 严重 | 影响核心功能 |
| Session失效 | 绿盟登录态过期 | 高 | 需重新登录 |
| 渠道故障 | 连续3次推送失败 | 高 | 渠道本身不可用 |

### 3.3 事件分级

| 级别 | 含义 | 响应要求 | 事件类型 |
|------|------|---------|---------|
| CRITICAL | 立即处理，影响业务 | 实时推送 + 可能需电话通知 | 采集失败、数据库异常、规则过期 |
| WARNING | 次日处理，暂不影响业务 | 汇总推送 | 规则无匹配、新包类型发现、磁盘空间低 |
| INFO | 记录即可，不一定推送 | 日志记录 | 服务重启（自动恢复） |

## 4. 功能设计

### 4.1 配置入口

「系统设置」→「系统事件通知」

```
系统事件通知
├── 启用开关 [ ]
├── 通知渠道 [下拉：选择已配置渠道]
├── 事件类型（多选）
│   ├── [ ] 采集失败（实时）
│   ├── [ ] 推送失败（实时）
│   ├── [ ] 规则无匹配（汇总）
│   ├── [ ] 规则过期（汇总）
│   ├── [ ] 新包类型发现（汇总）
│   ├── [ ] 服务重启（实时）
│   ├── [ ] 磁盘空间低（实时）
│   ├── [ ] 数据库异常（实时）
│   ├── [ ] Session失效（实时）
│   └── [ ] 渠道故障（汇总）
├── 实时推送：CRITICAL级别事件立即发送
└── 汇总推送：每日09:00发送WARNING级别事件摘要
```

### 4.2 去重/聚合策略

避免同一事件重复告警：

| 事件类型 | 去重窗口 | 聚合策略 |
|---------|---------|---------|
| 采集失败 | 1小时内 | 同产品多个URL失败聚合为1条 |
| 推送失败 | 5分钟内 | 同一目标连续失败聚合为1条 |
| 规则无匹配 | 按周期 | 不聚合，需每次都告 |
| 服务重启 | 1小时内 | 不聚合，每次都告 |
| 磁盘空间低 | 30分钟 | 不聚合，阈值突破才告 |

### 4.3 服务重启特殊处理

| 重启类型 | 是否通知 | 说明 |
|---------|---------|------|
| 手动重启（运维主动操作） | 否 | 主动操作无需告警 |
| 异常崩溃重启 | 是（CRITICAL） | 进程异常退出，需立即关注 |

识别方式：检测启动时前一条日志是否是正常shutdown。

### 4.4 消息格式

**实时告警**（采集失败/推送失败/服务重启）：
```
【CRITICAL】绿盟监控事件告警

事件：采集失败
产品：XXX
URL：https://xxx
时间：2026-05-24 10:30:00
详情：HTTP 503 - Service Unavailable
操作建议：请检查目标服务状态或网络连通性
```

**汇总报告**（规则无匹配/规则过期/新包类型）：
```
【绿盟监控】系统事件日报

汇总周期：2026-05-23 00:00 ~ 2026-05-24 00:00

● 规则无匹配（3条）
  - 客户A：规则「XXX」连续3周期无匹配，请检查订阅路径
  - 客户B：规则「YYY」连续5周期无匹配，请检查订阅路径

● 新包类型发现（1条）
  - 新增：产品A / V6.0.9 / rpm

● 规则过期（1条）
  - 客户C：规则「ZZZ」已于2026-05-23过期，推送已停止

处理建议：以上事件请登录绿盟监控平台处理
```

## 5. 技术实现要点

### 5.1 事件采集点

| 事件 | 采集位置 | 代码位置 |
|------|---------|---------|
| 采集失败 | `nsfocus.py` `_collect_url` 异常处理 | 捕获 `Exception` → 记录 + 触发事件 |
| 推送失败 | `router.py` `_send_immediate` / `mark_rollback_pending` | 发送失败或撤回时触发 |
| 规则无匹配 | `change.py` `get_new_for_subscription` | 连续周期计数，达标触发 |
| 规则过期 | `change.py` `get_new_for_subscription` 入口检查 | `valid_until` 过期返回空 |
| 新包类型发现 | `system_routes.py` `update_source` | 产品树新增节点时触发 |
| 服务重启 | `run.py` 启动时检测 | 检查前一条日志是否为正常shutdown |
| 磁盘空间低 | `scheduler.py` 或独立健康检查任务 | 定时检查 `/` 分区使用率 |
| 数据库异常 | `database.py` 异常处理 | 捕获 sqlite3 异常，触发事件 |
| Session失效 | `nsfocus.py` 登录检测 | 检测到登录失败时触发 |
| 渠道故障 | `router.py` 推送计数 | 连续失败N次触发 |

### 5.2 事件队列

系统事件统一进队列（可复用现有 `DIGEST_QUEUE` 或新建 `EVENT_QUEUE`），由独立 worker 处理。

### 5.3 配置存储

`settings` 表或独立 `system_event_config` 表：

```sql
CREATE TABLE system_event_config (
  id INTEGER PRIMARY KEY,
  enabled BOOLEAN DEFAULT FALSE,
  channel_id INTEGER REFERENCES notification_channels(id),
  event_types TEXT,  -- JSON数组 ["collect_failed", "push_failed", ...]
  quiet_start TEXT,  -- 静默时段开始（可选）
  quiet_end TEXT,    -- 静默时段结束（可选）
  created_at DATETIME,
  updated_at DATETIME
);
```

### 5.4 依赖关系

- 事件触发点采集（新增）

## 6. 优先级排期

### Phase 1：MVP（核心告警）
实现P0全部6个事件类型的触发和通知，作为系统可观测性的基础保障。

### Phase 2：扩展监控
实现P1事件类型，完善系统健康监控覆盖。

### Phase 3：增强能力
- 服务重启区分手动/异常
- 静默时段配置
- 告警升级策略（如连续N次告警未处理则升级电话通知）

## 7. 开放问题

1. **规则无匹配的周期阈值**：默认3个周期是否合适？还是需要可配置？
2. **汇总推送时间**：每日09:00是否合适？是否需要可配置？
3. **告警内容是否需要包含历史上下文**：如"这是本周第N次采集失败"方便定位重复问题？
4. **系统事件通知是否需要回复确认机制**：运维收到告警后是否需要确认已处理？
5. **历史事件查询**：管理员是否需要查看历史事件记录（事件日志）？
