# 升级包描述关键词高亮功能设计

> 版本：v1.0
> 日期：2026-05-28
> 状态：待开发

---

## 1. 背景与目标

绿盟升级包描述文本（`description_raw`）中包含大量操作提示信息，涵盖前置版本要求、重启设备/引擎提醒、业务影响警告等关键内容。推送通知中外发原文时，这些关键信息淹没在长文本中，管理员难以快速识别升级风险。

**目标**：在通知外发（企业微信/钉钉/飞书/邮件）和 Web 前端展示时，对描述文本中的关键段落做视觉高亮（加粗/标签/颜色），帮助管理员快速判断升级风险等级。

---

## 2. 高亮分类与关键词规则

基于 709 条升级包描述文本数据分析，提炼出以下高亮类别，按优先级分三层：

### 🔴 P0 — 影响升级是否成功（版本约束不满足则无法升级）

| 类别 | 关键词模式 | 示例 | 高亮样式建议 |
|------|-----------|------|------------|
| **前置版本** | `【前置版本】` + 版本号 | `【前置版本】6081.69905以及以上` | 加粗 + 红底标签 |
| **系统版本要求** | `请在XX版本及以上版本进行升级` | `请在64位系统版本V6.0R09F00.29386582及以上版本进行升级` | 加粗 |
| **升级前确认** | `升级前请确保当前系统版本` | `升级前请确保当前系统版本等于或高于V6.0.6.1.37100` | 加粗 |
| **系统版本条件** | `系统版本为XX及以上` | `系统版本为V5.6R63及以上` | 加粗 |
| **引擎版本约束** | `引擎版本为/大于/变更为` | `仅支持引擎版本为V5.6R11F08版本上进行升级` | 加粗 + 橙色 |
| **型号限制** | `不支持XX型号/设备` | `不支持海光设备升级` | 加粗 + 红底 |
| **配套补丁** | `需配合XX补丁包` | `需配合前置升级包使用` | 黄色标签 |

### 🟠 P1 — 影响升级操作体验（有重启/业务中断风险）

| 类别 | 关键词模式 | 示例 | 高亮样式建议 |
|------|-----------|------|------------|
| **重启设备** | `重启设备` / `自动重启设备` | `升级后需重启设备以使升级包生效` | 橙色 |
| **重启引擎** | `重启引擎` / `自动重启引擎` | `升级会重启引擎，请在业务空闲时操作` | 橙色 |
| **无重启** | `无需重启设备和引擎` | `升级后无需重启设备和引擎` | 绿色（正面） |
| **网络中断** | `网络中断` / `会话中断` / `流量中断` | `升级过程中会引起网络中断` | 橙色 |
| **业务空闲提醒** | `请在业务空闲时升级` | `请在业务空闲时升级，升级过程中请勿断电` | 黄色 |
| **升级耗时** | `耗时约X~Y分钟` | `根据设备型号不同，耗时约15~30分钟` | 灰色辅助 |
| **禁止强制重启** | `请不要强制硬重启设备` | `升级过程中请不要强制硬重启设备` | 红色 |
| **WEB不可用** | `WEB短时无法访问` | `升级过程中WEB短时无法访问，升级时间10分钟以上` | 黄色 |

### 🔵 P2 — 提示性信息（辅助判断升级后状态）

| 类别 | 关键词模式 | 示例 | 高亮样式建议 |
|------|-----------|------|------------|
| **全量升级包** | `全量升级包` | `本升级包为全量升级包，升级后固件版本不变` | 蓝色 |
| **增量升级包** | `增量升级包` | `本升级包为增量升级包，升级后固件版本不变` | 蓝色 |
| **固件版本变化** | `固件版本不变` / `固件版本变更为` | `升级后固件版本不变，引擎版本变为V5.6R11F08SP01` | 普通 |
| **引擎版本变化** | `引擎版本不变` / `引擎版本变更为` | `升级后固件版本不变，引擎版本变更为V5.6.9.91` | 普通 |
| **规则版本变化** | `规则版本变为` | `入侵防护特征库版本变为5.6.11.44743` | 普通 |
| **初始化中禁止升级** | `正在初始化` | `如系统正在初始化(如：系统开机，系统升级，系统还原)` | 黄色 |
| **升级回退** | `升级回退` / `还原` | `升级失败后支持回退` | 灰色辅助 |

## 3. 实现方案

### 3.1 实现位置

```
src/notifiers/base.py
```

所有外发通知的消息体均通过 `_format_markdown_body()` 和 `_format_markdown_bodies()` 生成，在 `description_full` 输出前调用高亮处理函数。

Web 前端（`src/web/templates/index.html`）如需同步高亮，可在 JS 层复用同一套关键词规则。

### 3.2 核心函数设计

```python
# 新增函数
def _highlight_upgrade_keywords(text: str) -> str:
    """
    对升级包描述文本中的关键段落进行 markdown 高亮标记。
    
    高亮策略：
    - P0 风险词：加粗 + **（部分 IM 渲染为红色）
    - P1 风险词：加粗（部分 IM 渲染为橙色）
    - P2 提示词：普通文字，不做处理
    
    所有匹配使用精确短语匹配，避免误高亮正文内容。
    """
    ...
```

### 3.3 Markdown 高亮样式选择

考虑到企业微信/钉钉/飞书均支持标准 Markdown，选用以下样式：

| 级别 | 样式 | 示例 | 说明 |
|------|------|------|------|
| P0 | `**文本**`（加粗） | `**【前置版本】6081.69905**` | 各 IM 均支持，视觉突出 |
| P1 | `**文本**`（加粗） | `**重启设备**` | 同上，橙色程度弱于红色 |
| P2 | 不处理 | — | 辅助信息无需高亮 |

> **注**：如 IM 平台支持背景色（飞书支持 HTML `<mark>`），可进一步区分。未避免兼容性问题，优先使用加粗方案。

### 3.4 匹配策略

- **短语精确匹配**：使用完整短语（如 `重启设备`、`重启引擎`、`无需重启设备和引擎`），避免单词级别误匹配
- **优先级覆盖**：P0 > P1 > P2，同一文本同时匹配多个类别时，保留最高优先级
- **顺序处理**：先替换 P0 关键词，再处理 P1，避免重叠替换
- **正则边界**：版本号使用 `[\d\.R]+` 模式，防止截断

### 3.5 外发入口确认

所有通知渠道（企业微信/钉钉/飞书/邮件）均通过以下路径调用：

```
_format_markdown_body(msg)  →  line 253  msg.description_full
_format_markdown_bodies() →  调用 _format_markdown_body()
```

高亮函数在 `description_full` 输出前调用一次即可，**不影响元数据行**（文件/包版本/MD5/发布时间）。

## 5. 验收标准

1. 触发推送时，带 `description_full` 的通知消息中，版本约束/重启/业务影响关键词已做加粗处理
2. 高亮标记仅作用于 `description_full` 区域，元数据行（文件/包版本/MD5）不受影响
3. 同一文本被多个关键词覆盖时，仅应用最高优先级样式
4. 高亮函数不修改原始 `description_raw`，仅在渲染时做临时转换
5. Web 前端展示描述文本时，关键信息视觉突出，管理员可快速定位升级风险
