## 3. 核心设计决策

### 3.1 采集统一抽象

```
BaseCollector (ABC)
├── source_type: str
├── collect(source_id, cookie) → list[UnifiedContentItem]
└── check_health(cookie) → CollectorHealth

NsfocusCollector(BaseCollector)
├── session: requests.Session          # 采集用 session
├── _discover_session: requests.Session # 发现用独立 session（隔离污染）
├── PRODUCTS 字典驱动（可扩展至任意数量产品）
├── 标准型：3级遍历（版本→包类型→table）
├── 递归型：最多4层深（遇table停止）
├── Quick 模式：页面 hash 对比，变化才全量解析
├── 发现模式：递归遍历发现 paths + package_types
└── 描述结构化提取
```

### 3.2 通知统一格式

```python
@dataclass
class NotificationMessage:
    title: str
    summary: str
    product_name: str
    version_branch: str
    package_type: str
    file_name: str
    package_version: str
    md5_hash: str
    description_full: str        # 完整描述（含结构化摘要）
    urgency: str                 # normal | high | critical
    source_url: str
    download_url: str
    attachment: Optional[bytes]  # 附件内容（≤10MB）
    attachment_name: str
```

每个 Notifier 实现 `send(message, config) → DeliveryResult`。

### 3.3 订阅规则 — 链路径匹配（v2.0）

```
filter_conditions.chains[]:
  { chain: ["WAF", "标准正式版", "V6.0.8", "规则"], match: "leaf" }
  { chain: ["IPS", "V6.0.9"], match: "subtree" }

chain 反查机制：
  snapshot.source_url
      ↓
  content_sources.package_type.paths[] → 找匹配 URL
      ↓
  chain → ["WAF", "标准正式版", "V6.0.8", "规则"]
```

运行时通过 scheduler 启动时构建的 `_url_chain_cache` 反查，无需 snapshot 表增加列。

### 3.4 推送模式（v2.0）

```
即时模式：delay_days（0~30天）
  - 新包入延迟队列，到期后发
  - 包撤回时取消队列中的条目

汇总模式：window_config（周几 + 时间段）
  - 窗口内合并推送
  - 窗口外入 delayed_queue 等下一个窗口开启
```

回退通知：不受 delay/window/静默限制，即时推送。

### 3.5 Session 隔离策略

```
Scheduler 采集（全量/增量/快速）
  → _session（采集用 requests.Session）
  → 与 discover 完全隔离，互不干扰

discover_products() / discover_package_types()
  → _discover_session（独立的 requests.Session）
  → 撞到 /upLic → RedirectToLicenseError → 跳过
  → 被污染的只是 discover session，无影响
```

Session 按 `purpose`（用途）和 `collect_mode`（采集模式）分组：

| purpose | collect_mode | 说明 |
|---------|-------------|------|
| discover | (空) | 发现用，被污染无所谓 |
| collect | standard | 标品采集（`downloads/id`） |
| collect | vm | 虚拟化采集（`downloadsVm/id`） |

### 3.6 产品管理 ✅ 已完成（2026-05-15）

Web UI 产品管理面板（系统设置页）提供：
- **自动发现**：输入 URL → 递归遍历 → 解析全部产品链接 → 入库
- **开关管理**：每个产品独立开关，`is_active=1` 才参与采集
- **手动注册**：填写名称 + 入口 URL，可选策略
- **策略检测**：有 session 时自动分析产品首页结构，推荐 standard/recursive

### 3.7 系统事件通知（v2.0）

```
事件类型：
  - collection_summary：采集完成（有声/有错/空跑三态）
  - session_error：Session 异常或健康检查失败
  - log_error：日志扫描异常

通知分发：
  event_handler.emit_*()
    → is_event_enabled(event_type) 检查开关
    → get_notify_channel() 获取渠道
    → notifier.send() 发送
    → log_event() 记录到 system_event_log
```

系统事件可独立于订阅规则配置通知渠道。

## 4. 技术选型

| 选择 | 理由 |
|------|------|
| SQLite | 单机部署，无需额外数据库进程，数据量小(<100MB)，WAL 模式并发安全 |
| Flask | 轻量，满足 Web 面板需求，无过度抽象 |
| APScheduler | Python 定时任务，比 cron 更灵活，支持内存 + 持久化混合模式 |
| cryptography | AES-256-GCM 加密，Python 官方推荐 |
| bcrypt | 密码哈希 |
| PyJWT | JWT 认证 |
| requests | HTTP 客户端，支持 Session/Cookie 管理 |
