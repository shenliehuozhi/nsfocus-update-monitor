## 表设计要点

### 1. 加密字段（AES-256-GCM）

以下字段加密存储：
- `user_sessions.cookie_value`
- `channels.config`（包含 webhook_url / smtp_password）

密钥存储在环境变量 `MONITOR_SECRET_KEY`。

### 2. JSON 字段

SQLite 原生支持 JSON（>=3.38），以下字段使用 TEXT 存储 JSON：
- `customers.owned_products`: `["WAF:V6.0.9", "WAF:V6.0.8", "IPS:5.6.11"]`
- `subscription_rules.filter_conditions`: 结构化过滤条件（chains + keywords + urgency）
- `subscription_rules.window_config`: `{days:[0,1,2,3,4], start:"09:00", end:"18:00"}`（nullable）
- `snapshots.description_parsed`: `{added:[], modified:[], deleted:[], other:"", ...}`
- `content_sources.package_type`: 发现后的完整链信息
- `channels.config`: 渠道配置

### 3. filter_conditions 结构（v2.0）

```json
{
  "chains": [
    { "chain": ["WAF", "标准正式版", "V6.0.8", "规则"], "match": "leaf" },
    { "chain": ["IPS", "V6.0.9"], "match": "subtree" }
  ],
  "keywords": ["漏洞"],
  "urgency": ["high"]
}
```

- `chains` 为空或不存在 = 匹配全部（向后兼容旧数据）
- `match=leaf`：精确匹配具体包类型链
- `match=subtree`：订阅该节点下所有后代

### 4. 索引

```sql
CREATE INDEX idx_snapshots_source ON snapshots(source_id);
CREATE INDEX idx_snapshots_product ON snapshots(product_name, version_branch);
CREATE INDEX idx_snapshots_status ON snapshots(status);
CREATE INDEX idx_snapshots_md5 ON snapshots(md5_hash);
CREATE INDEX idx_delivery_snapshot ON delivery_log(snapshot_id);
CREATE INDEX idx_delayed_queue_status ON delayed_queue(status, push_after);
CREATE INDEX idx_audit_user ON audit_log(user_id, created_at);
```

### 5. 表关系约束

- `snapshots`: (source_id, product_name, version_branch, package_type, md5_hash) 组合唯一
- `delayed_queue`: snapshot_id + rule_id 唯一
- `delivery_log`: snapshot_id + channel_id + rule_id 唯一（v1.5+ 加 rule_id 去重）
- `customers.name`: UNIQUE

### 6. 关键设计决策

**Session 心跳状态**（`user_sessions.heartbeat_status`）：
- `ok`：最近一次心跳健康
- `过期`：Session 在绿盟侧已过期（302 重定向）
- `污染`：Session 上下文被 upLic/Vm 格式污染（返回 downloadsVm/id）
- `错误`：网络错误或其他异常

**chain 反查机制**：snapshot 通过 `source_url` 在 `content_sources.package_type.paths[]` 中反查 chain，运行时获取，无需 snapshot 表增加列。

**window_config 语义**：
- 即时模式：`window_config` 为 null，`delay_days` 控制延迟
- 汇总模式：`window_config` 有值，`delay_days` 被忽略

**system_event_config.event_types**：
- 空数组 = 全部启用
- 非空 = 只启用列表中的事件类型

### 7. 已移除字段

- `customers.warranty_start` / `warranty_end`：维保期字段已移除（2026-05-24）
- `subscription_rules.delay_hours`：改为 `delay_days`（2026-05-23）
- `subscription_rules.delay_strategy`（reset/append）：UI 已移除，仅 DB 列保留向后兼容
- `subscription_rules.min_interval_hours`：UI 已移除，仅 DB 列保留向后兼容

## 数据清理

- 快照数据保留 180 天：`DELETE FROM snapshots WHERE last_seen_at < datetime('now', '-180 days') AND status = 'rollback'`
- 推送日志保留 365 天
- 审计日志保留 365 天
- 清理任务作为定时任务每周执行
