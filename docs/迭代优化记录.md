## v1.3 — 订阅规则字段精简（2026-05-23）

### 背景

即时模式和汇总模式的字段语义混乱，部分字段是死代码。

### 改动

| 字段 | 即时模式 | 汇总模式 |
|---|---|---|
| delay | 保留（天） | 移除 |
| 策略（reset/append） | 移除（死代码） | 移除（死代码） |

| 窗口策略 | 独立checkbox | 独立checkbox |

### 设计决策

- 即时模式：`delay_days` 独立计时，新包不取消旧计时器

- `delay_strategy`（reset/append/window）和 `min_interval_hours` DB列保留，向后兼容

## v1.6 — 代码质量 + 订阅条件细化（2026-05-21）

### 代码质量

- dashboard 加 `@require_auth` 装饰器
- 全局裸 except → `except Exception` + 日志
- 白名单校验：所有用户输入做格式校验

### 订阅条件细化（核心改进）

**问题**：原 `filter_conditions` 为扁平三字段，无法精确匹配"产品→子分类→版本→包类型"层级结构。

**解决方案**：引入 `chains[]` 结构，支持 leaf/subtree 两种匹配模式：

```json
{
  "chains": [
    { "chain": ["WAF", "标准正式版", "V6.0.8", "规则"], "match": "leaf" },
    { "chain": ["IPS", "V6.0.9"], "match": "subtree" }
  ]
}
```

- **leaf**：snap_chain 与 rule_chain 完全相等（订阅具体包类型）
- **subtree**：snap_chain 以 rule_chain 为前缀（订阅中间节点下全部）

**实现**：通过 `source_url` 在 `content_sources.package_type.paths[]` 中反查 chain，无需 schema 改动。运行时 `_get_chain(source_id, source_url)` 从缓存查询。

## v1.7 — 稳定性 + 系统事件通知（2026-05-25）

### 问题1：database is locked

**根因**：4个 Python 进程同时写 DB，SQLite 内部只能有一个写操作，WAL 写锁竞争。

**修复**（commit fbeb455）：
- `_write_lock` 从 `Lock()` 改为 `RLock()`（解决重入死锁）
- WAL checkpoint 从 `PASSIVE` 改为 daemon thread 每5分钟 `TRUNCATE`
- `busy_timeout` 从默认值提升到 10s
- `execute()` 增加 3 次重试机制（100/200ms 递增）

### 问题2：系统事件通知缺失

**根因**：无新包时采集结束无信号，运维无法感知系统健康状态。

**修复**：
- 采集结束写入 `push_summary` 到 `system_event_log`
- 系统事件配置中可独立开关是否发送企微通知
- 企业微信渠道跳过 `_format_markdown_body`，直接发送纯文本

### 其他改进

- 调度器全局开关：调试时可关闭采集
- 心跳函数 DB 操作包裹 try-except，防止崩溃
- 客户名称唯一性校验（API层友好提示）
- DELETE 尾部斜杠 404 修复

## v2.0（规划中）— 采集性能优化

### 目标

解决"一次采集触发~1606次HTTP请求"导致的峰值QPS过高、目标服务器压力大问题。

---

### 策略1：Last-Modified 变化记忆

**现状**：每轮采集对所有803 URL发 HEAD 请求，确认ETag/Last-Modified变化后才发 GET。

**优化**：记忆上次 GET 时间，下次只发 HEAD。对未变化的URL直接跳过 GET。

**效果**：803 URL → 约401 HTTP请求/轮（减半）。

---

### 策略2：采样 v2（概率 skip）

**现状**：所有 URL 每轮必扫，部分旧 URL（>365天）的页面内容多年未变，仍每次发完整请求。

**优化**：按 URL 价值分层采样，减少无效请求：

```python
# 核心公式
prob = (
    0.15 × e^(-age/365)        # 年龄因子（半衰期365天）
  + 0.50 × delivery_score      # 推送频率因子
  + 0.25 × change_recency     # 近期变化因子
  + 0.50 × never_delivered_bonus  # 从未推送bonus
)
```

**关键设计**：
- `must_scan_now()`：连续 skip 超标或从未推送的 URL 强制扫
- 变化后首次扫描不参与 skip 计数
- `expected_interval_days`：从历史数据估算下次更新时间

**效果**：低价值 URL 降到 2% 概率，高价值 URL 保持 77%+。

---

### 策略3：桶轮转（分散扫描）

**现状**：定时任务触发 → 一次性遍历全部803 URL → 峰值~40 req/s。

**优化**：将803 URL按 product 分组，轮转分配到60桶（每桶约13个URL）。每5分钟触发一次，只处理一个桶。

**效果**：峰值QPS从~40降至~0.3。

---

### 策略4：联合优化（桶轮转 × 采样v2 × 优先级队列）

采样v2（决定要不要扫）、桶轮转（决定什么时候扫）、优先级队列（高频繁URL额外加速）是**三层正交优化**，叠加效果：

| 阶段 | 每轮HTTP请求 | 高频繁URL实时性 |
|---|---|---|
| 改造前（一次性全扫，无采样） | 1606 | N/A |
| 桶轮转（全量扫桶内URL） | ~27 | 最多等4小时 |
| 桶轮转 + 采样v2（50% skip） | ~14 | 最多等4小时 |
| 桶轮转 + 采样v2 + 优先级队列 | ~20 | 每5分钟扫（12次/4小时） |

**优先级队列设计**：

```python
HIGH_FREQ_THRESHOLD = 80  # priority >= 80 的URL视为高频繁

def scheduler_tick():
    # 高频繁URL：不受桶限制，每tick都处理
    high_freq_urls = [u for u in urls if u.priority >= HIGH_FREQ_THRESHOLD]
    for url in high_freq_urls:
        scan_url(url, reason='high_freq')
        adjust_priority(url, delta=+5)  # 有变化则升高

    # 普通URL：按桶轮转 + 采样v2
    ...
```

**优先级动态调整**：
- 发现变化 → `priority += 5`（最高100）
- 无变化且skip → `priority -= 2`（最低10）

**字段复用**：`collection_urls.priority` 同时服务采样v2和高频繁判断，无需新增字段。

**峰值QPS从~40降至~0.5**，高频繁URL实时性从"最多等4小时"提升到"每5分钟扫"。

```
scheduler_tick() 每5分钟触发
    → 获取当前桶URL（约13个）
        → must_scan_now() == True → 强制扫
        → prob >= random() → 扫
        → prob < random() → skip
    → 完成当前桶 → 更新 checkpoint
    → 发现新包 → 立即触发检测+推送
```

---

### 数据结构

新建 `collection_urls` 表，统一存储采样v2状态 + 桶轮转状态：

```sql
CREATE TABLE collection_urls (
    id INTEGER PRIMARY KEY,
    source_id INTEGER NOT NULL,
    url TEXT NOT NULL,
    url_hash TEXT NOT NULL UNIQUE,

    -- 桶轮转相关
    scan_bucket INTEGER DEFAULT 0,    -- 分桶号 0-59
    last_scan_at TEXT,
    last_scan_round INTEGER DEFAULT 0,

    -- 采样v2相关
    change_count INTEGER DEFAULT 0,
    last_changed_at TEXT,
    consecutive_skips INTEGER DEFAULT 0,
    expected_interval_days INTEGER DEFAULT 30,
    scan_priority_score REAL DEFAULT 0,

    -- 共用
    scan_count INTEGER DEFAULT 0,
    priority INTEGER DEFAULT 50,
    created_at TEXT,
    updated_at TEXT
);
```

## v2.1（规划中）— 单线程写入者模式

### 问题

当前 `_write_lock` 锁粒度太大：collector 持有锁写大量数据时，心跳、日志扫描等线程全部 blocked，导致 `database is locked`。

### 方案

将 DB 写操作统一到单个 writer 线程，collector 的写操作改为 enqueue：

```
Collector → enqueue_write(WriteTask) → Queue → Writer Thread → execute()
```

**优点**：
- 消除多线程竞争写锁
- WAL checkpoint 无需 TRUNCATE（主动推送即可）
- DB 写入顺序可预测

**实现步骤**：
1. 新增 `queue.Queue` + `WriteTask`
2. writer 线程从队列取任务执行
3. `execute()` 路由到 `enqueue_write()`
4. 修正 nsfocus.py 两处绕过 `_write_lock` 的 `execute()`
5. 删除 `_write_lock`，改用 Queue

## v2.2（规划中）— 采集链路可视化

### 目标

前端实时展示各 phase 状态、耗时、进度，支持历史对比。

### 数据结构

```sql
CREATE TABLE collection_runs (
  id INTEGER PRIMARY KEY,
  mode TEXT NOT NULL,
  started_at TEXT NOT NULL,
  finished_at TEXT,
  status TEXT NOT NULL,
  total_new INTEGER DEFAULT 0,
  total_rollback INTEGER DEFAULT 0,
  phases TEXT,            -- JSON: [{name, status, duration_ms, writes}, ...]
  product_timings TEXT,   -- JSON: [{name, duration_ms, new_count}, ...]
  errors TEXT             -- JSON: [{phase, msg}, ...]
);
```

### Phase 梳理（完整链路10 phase）

| # | Phase | 典型耗时 | DB写操作 |
|---|---|---|---|
| 0 | startup | <100ms | system_settings（1次） |
| 1 | session_check | <1s | sessions（过期时才写） |
| 2 | source_bootstrap | <1s | content_sources（偶尔） |
| 3 | collection | 数分钟 | 高频：健康度更新、逐包写入snapshots |
| 4 | detection | 秒级 | snapshots 新增/回滚写入 |
| 5 | notification | 秒级~分钟 | delivery_records |
| 6 | delayed_queue | 秒级 | 更新 delivery_records 状态 |
| 7 | pkg_type_refresh | 分钟级 | 更新 package_type_discovered |
| 8 | push_summary | 秒级 | 少量统计写入 |
| 9 | system_event | 秒级 | system_events |
