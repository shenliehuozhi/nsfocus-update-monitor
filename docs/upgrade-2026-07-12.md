## 1. 订阅规则 UI + 邮件 fallback(433aea3 ~ a377f0e)

### 1.1 列表模板列显示修复 — commit 433aea3

#### 问题

订阅规则列表的「通知模板」列硬编码用 `r.template` 取值,但当前实现是 **per-channel**(每条 rule_channels 各自的 template),导致**所有规则都显示「A · 完整」**,跟实际配置(用户可能选 B/C/D)完全脱节。

**实测对比**(用户原话描述):

| 规则 | 用户实际配置 (DB) | 列表显示(修复前) | 列表显示(修复后) |
|---|---|---|---|
| rule 1004 企微 | per-channel `brief` (C) | A · 完整 ❌ | C · 极简 ✓ |
| rule 1005 钉钉 | per-channel `strip` (B) | A · 完整 ❌ | B · 智能裁剪 ✓ |
| rule 1006 飞书 | per-channel `feishu_full` (D) | A · 完整 ❌ | D · 飞书全量 ✓ |
| rule 1007 邮件 | email(不走模板) | A · 完整 ❌ | 邮件 HTML ✓ |

#### 根因

`src/web/templates/index.html` 列表渲染用 `({full:'#e8f0fe',...})[r.template||'full']` 一刀切取标量 template,完全没看 per-channel 的 `rule_channels.template`。

#### 修复

- 新增 `renderRuleTemplateSummary(r, channels)`,按规则每条 rule_channel 单独 fallback
- 优先级:rule_channels.template → rule.template dict[str(cid)] → CHAN_DEFAULT_TPL[type] → full
- **飞书特殊归一**:`chType==='feishu' && tpl!=='feishu_full'` 强制设为 `feishu_full`(底层 feishu.py `_build_post` 走 post 格式,不读 `_template`,4 模板对飞书都退化为 D)
- 飞书新增规则时:打开弹窗 → 添加飞书渠道 → 立即选中 D(避免空白),saveRule 提交 `template=feishu_full` 而非空

#### 验证

- 浏览器实测 4 种 case(per-channel / 飞书归一 / 邮件 HTML / 全角字典) 全部正确
- 后端 35/35 test_router_decisions 通过(模板 fallback 链测试)

### 1.2 邮箱 + 附件大小 per-customer fallback — commit 8609aff

#### 问题

「订阅规则 → 添加邮件渠道 → 收件人/附件大小留空 → 推送失败」的真实场景:用户在「客户」档案里已经配了邮箱 + 附件大小,但还要在每条订阅规则里再手填一遍。

**3 级 fallback 链断裂在中间环节**:

| 字段 | 收件人 fallback | 附件大小 fallback |
|---|---|---|
| per-channel(rule_channels) | ✓ | ✓ |
| 全局(subscription_rules.customer_emails) | ✓ | ✗ 不查客户 |
| **per-customer(customers)** | ✗ **不查客户** | ✓ |
| 默认 | 不设 → 邮件空 | 10MB |

**L3 邮箱链路**完全缺失 — 用户在「客户」配了邮箱,规则里留空,后端就拿空字符串,邮件发不出去。

#### 修复

- **后端** `src/notifiers/router.py:232-258` 加 per-customer 邮箱 fallback(完全对称附件大小的实现)
- **前端** 2 字段改 B 模式:input.value 留空,placeholder 展示客户档案值;saveRule 写 0,后端自动 fallback
- **前端** 「关联客户」下新增一行 `📎 客户档案附件大小: 15 MB(来自客户档案)` — 永久可见,客户改了档案值也能感知

**B 模式 vs A 模式权衡**:
- A:预填客户档案值 → saveRule 写死 → 客户改档案不影响规则
- B:value 留空,placeholder 提示 → 客户改档案,所有规则自动跟 ✦

选 B(用户偏好,与「客户档案是默认值」的语义一致)

#### 验证

- 4 个新 test_email_recipient_* 覆盖 L1/L2/L3/L4 fallback 优先级
- 真实数据 rule 1007(rule_channels 留空,客户档案有 email/15MB)→ push 走 L3 成功
- 35/35 test_router_decisions + 50/50 test_diff_parser 全过

## 2. 规则 Diff 工具鲁棒性(b2b75da ~ 563d1f7)

### 2.1 IPS Diff 工具 0 条 — commit b2b75da

#### 问题(用户报告)

**IDPS 规则库 Diff**: `https://update.nsfocus.com/update/listNewipsDetail/v/rule5.6.11_v2` FROM `2.0.0.32720` TO `2.0.0.32939`,中间 2 个版本,**新增 0 / 修改 0 / 删除 0**。

#### 根因

`parse_rules` 用 `_slice_block(text, '新增规则：', ...)` 搜**全角冒号**做切块。但实测 vendor 真实页面用**半角冒号** `新增规则:`(无全角),`find` 找不到 → 返回空字符串 → parse 0 条。

#### 实测复现

- 修复前:`parse_rules(2.0.0.32720 desc_raw len=12088)` → `added=0, updated=0`
- 修复后:`added=49, updated=32` (跟 vendor 页面描述一致)
- diff stats 修复前: `pure_new=0, pure_updated=0, pure_deleted=0`
- 修复后: `pure_new=117, pure_updated=8, pure_deleted=79`(含 2 个中间版本)

#### 修复

`parse_rules` 内**预处理** description_raw,半角冒号 marker 归一为全角(不动 `_slice_block` 通用接口,WAF 不受影响)。

### 2.2 IPS marker 后缀多种符号兼容 — commit 535417f

半角冒号修了,但 vendor marker 后面还能跟什么?(实测 5 种变体 + 推断 2 种):

| 后缀 | 含义 | 改前兼容? |
|---|---|---|
| `:` / `：` | 冒号(全/半角) | ✓ |
| `:.` | 冒号+点+换行 | ✗ |
| `
` | 没冒号,直接换行 | ✗ |
| `
无

| ` ` 空格 | 空白 | ✗ |
| `	` 制表符 | 控制符 | ✗ |
| `` 回车 | 控制符 | ✗ |

扫整个 description_raw,marker 后紧跟的 1+ 个分隔符 `[:：\s

### 2.4 WAF marker 编号前缀可选 — commit 563d1f7

#### 问题(用户)

> 「一、新增规则 那有没有可能 有时错写成一.新增规则 我觉你WAF也应该优化下 应该是以新增规则为关键字 而不是 一、新增规则」

旧 WAF `parse_rules_waf` 硬编码依赖「一、新增规则」「二、修改规则」等带**中文数字、**前缀的 marker。vendor 写「一.」「1.」「(一)」「无编号」都 0 条。

#### 设计

以**纯关键字**(`新增规则` / `修改规则` / `删除规则` / `其他修改` / `升级建议`)为 marker,**编号前缀可选**。regex 行首匹配,可识别 8 种格式(实测):

| vendor 写 | 旧版 | 新版 |
|---|---|---|
| `一、新增规则` | ✓ | ✓ |
| `一.新增规则`(用户提的「错写」) | ✗ | ✓ |
| `1. 新增规则` | ✗ | ✓ |
| `1、新增规则` | ✗ | ✓ |
| `(一)新增规则` | ✗ | ✓ |
| `（1）新增规则` | ✗ | ✓ |
| `1.新增规则` | ✗ | ✓ |
| `新增规则`(无编号) | ✗ | ✓ |

#### 顺带修了一个小 bug

旧版用 `len(cleaned) < 10` 粗判段是否「太短」,**会把只 1 条规则的段错误丢弃**。改用「剥段头第一行后,看规则行数=0 才丢」,更精确。

#### 验证

- 6 个新 test 覆盖 8 种编号格式 + 真实数据 dry-run
- 真实 WAF 数据 6.0.8.0.62275/5.6.10.45304 仍正确(3+0+0 / 6+9+0)
- 50/50 test_diff_parser 全过

### 2.5 前端 2 入口 URL kind 校验 — commit 33c4374

#### 问题(用户多轮纠正后)

> 「那你这实际上WAF /IDPS工具实际只是一个工具 而不是2个」
> 「B 那就不能两个工具,不同工具单独校验是否是对应WAF或者IDPS的URL么」

**前端现状**:`toolDiffRuleOpen`(IDPS 入口) + `toolDiffWafOpen`(WAF 入口) — 2 个入口但**只设默认值**,`sourceKind` 初值可被后端 fetch 时按 URL 路径族覆盖。

**真实风险**:用户在 IDPS 入口粘 WAF URL,后端静默按 WAF 处理,前端无任何提示。**入口的工具身份失效**。

#### 修复(选 B 方案)

- 2 个入口各带 `expectedKind` 参数
- `rdUrlValidate` 检测到 URL kind 跟 `expectedKind` 不匹配时:
  - **置灰「提取」+「强制刷新」按钮**
  - 显示红色警告 `🚫 入口跟 URL 不匹配`
  - 提示用户「请点 ← 返回切到对应工具,或改正 URL」
- `rdFetch` 提交前再校验一次,defense in depth
- 入口函数不再设 `_rd.sourceKind` 初值(后端 fetch 时返 source_kind 覆盖,前端不再需要)
- localStorage key 改统一 `rd_last_url`(不再按 sourceKind 分)

**2 个工具入口保留**(用户选的 B),但**真的**区分了工具职责,不再 silent 切换。
