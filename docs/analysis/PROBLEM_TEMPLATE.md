# 问题分析报告模板

> 每一次线上问题的根因分析都应写成此格式，存于 `docs/analysis/` 目录。
> 文件命名：`YYYY-MM-DD-问题简述.md`

---

## 一、问题描述

### 1.1 现象（Symptoms）
- 用户看到的具体表现
- 错误日志/告警信息
- 影响范围（哪些功能不可用、多少人受影响）

### 1.2 环境
- 操作系统 / 内核版本
- Python 版本
- SQLite 版本
- 部署方式（gunicorn / Flask内置服务器）
- 相关配置参数

### 1.3 时间线
```
HH:MM  用户反馈/告警触发
HH:MM  开始排查
HH:MM  发现根因
HH:MM  实施修复
HH:MM  验证通过
```

---

## 二、原因分析

### 2.1 初步假设
列出最初怀疑的方向及排除依据。

### 2.2 证据收集
- 日志分析（关键片段）
- 代码审查（相关函数/模块）
- 调试输出
- 验证测试

### 2.3 根因定位
用因果链的方式描述：
```
原因A → 导致 → 原因B → 导致 → 原因C → 导致 → 问题现象
```

**如果是多层嵌套，标注每一层的贡献度**：
- 根本原因（最深层的触发条件）
- 直接原因（最近一步的代码逻辑）
- 放大因素（让问题恶化的条件）

---

## 三、相关原理

### 3.1 技术背景
涉及的框架/库/系统的内部机制。

### 3.2 配置参数说明
相关配置项的含义、默认值、正确用法。

### 3.3 常见误区
这个知识点容易被误解的地方。

---

## 四、修复方案

### 4.1 修复代码
关键改动（完整 diff 或关键片段）。

### 4.2 验证方法
- 如何确认问题已解决
- 验证命令/步骤
- **必须包含实际执行的命令和输出**

**验证状态表**（每项都必须实际执行并记录结果）：

| 检查项 | 期望结果 | 实际结果 | 状态 | 执行时间 |
|--------|----------|----------|------|----------|
| | | | ✓/✗ | |
| | | | ✓/✗ | |
| | | | ✓/✗ | |

**验证命令记录**（复制粘贴自实际执行）：
```
$ <实际命令>
<实际输出>
```

> 验证必须实际执行并记录输出，口头声明"已验证"无效。

### 4.3 配置变更
所有配置参数的修改记录。
