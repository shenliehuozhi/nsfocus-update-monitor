# 重启不立即采集 —— 设计评估

> 状态: **讨论完成,未实施**。开会话做改动时直接接上。
> 日期: 2026-06-23

## 一、痛点

`start_scheduler()` 启动后,APScheduler 注册的 `nsfocus_collect` job 设了
`next_run_time = utcnow() + 10s`,导致**重启服务后 10 秒就立刻跑一次采集**。

用户希望:**重启不要立刻采集,按原周期计划跑**(周期不漂移)。

## 二、根因定位

**文件**: `src/core/scheduler.py`

**两处"启动立刻跑"的源头**:
1. `refresh_scheduler_jobs()` 1112-1115 行:`next_run_time=datetime.utcnow() + timedelta(seconds=10)`
2. `start_scheduler()` 1192-1197 行的 `_delayed_first_run` 后台线程 → 5 秒后调 `_check_first_run`
   (1239-1252 行)。`_check_first_run` 检查"是否有 source_url 已填充的活跃 snapshot",
   如果没有则触发 full scan。

**特别说明**: `_save_last_collect_at()` 函数(859-861 行)注释明确写
"last_collect_at is now read from MAX(last_collected_at) in content_sources"
—— 也就是说**"按上次采集时间推算 next_run"这个设计意图早就有了,只是没接到
`add_job` 的 `next_run_time` 参数上**。

## 三、设计方案(已讨论确认)

### 3.1 数据源:`MAX(content_sources.last_collected_at)`

**为什么用 content_sources 不用 snapshots**:
- `content_sources.last_collected_at` = 某个 source(产品/源)最后一次被采集完成的时间
- `snapshots.last_seen_at` = 某个具体升级包最后一次被发现/刷新
- 每次 source 采完都会调 `update_source_health(source_id, 'ok', utcnow().isoformat())`
  写入 `content_sources.last_collected_at`(见 scheduler.py 755/803/1043/1048 行)
- `MAX(content_sources.last_collected_at)` 语义 = "上一次采集结束时间",**与周期推算需求完全匹配**

**初始化时字段值**:
- 字段定义: `last_collected_at TEXT,`(snapshot.py:18,**无 DEFAULT**)
- `initial_sources.json` 不含此字段
- `create_source()` 不写此字段
- **全新部署时:NULL**

### 3.2 next_run 计算逻辑

```python
last_max = query("SELECT MAX(last_collected_at) FROM content_sources")
if last_max is None:
    # 全新部署:以启动时间作为基线,写库,避免下次重启又触发"立刻跑"
    now_iso = datetime.utcnow().isoformat()
    execute("UPDATE content_sources SET last_collected_at = ? "
            "WHERE last_collected_at IS NULL OR last_collected_at = ''", (now_iso,))
    last_max = datetime.utcnow()
next_run = last_max + timedelta(hours=interval_hours)
if next_run <= datetime.utcnow():
    # 该跑了(周期到了 或 服务挂太久,APScheduler misfire 补跑)
    next_run = datetime.utcnow() + timedelta(seconds=10)
# 否则 next_run 是未来时间,APScheduler 自动等

add_job('nsfocus_collect', trigger='interval', hours=interval_hours,
        next_run_time=next_run, ...)
```

**关键点**:
- 全空 → 写库 now + 算 4h 后跑(给用户留时间加 session/source)
- 正常 → 按"上次完成 + interval"算,**周期不漂移**
- 周期到了或挂太久 → 立即补跑(misfire 语义)

### 3.3 启动前置条件(避免"没 session 乱跑")

`run_now(mode='full')` 会真去绿盟官网请求页面,没 session 会被 302 踢回登录页/触发账号风控。

**实现**:不直接在 `start_scheduler` 里硬调 `run_now`,而是在 `add_job` 前判断:
- 没 active session(查 `user_sessions` 表 `status='active'`)→ `next_run = None`,job 注册但不触发
- 没 source → 同上

**配套**:Web "添加 session" 接口成功后,调一次
`_scheduler.reschedule_job('nsfocus_collect', trigger='interval', hours=interval, next_run_time=now+10s)`
→ 用户配完 session 立刻跑。"添加 source" 接口同理。
